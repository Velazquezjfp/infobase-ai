# Resolve documented anti-patterns in API and environment surfaces

## Description

The current-state docs explicitly flag two duplications that should be cleaned up before the closed-environment deployment, plus one minor health-check inconsistency. These are the only items in scope for sprint 1 under the input's directive: "check conflicts and potential errors from the documented api, db and environment under the folder /docs, some docs could point to antipatterns or redundancies".

**1. Duplicate `VITE_API_URL` and `VITE_API_BASE_URL`** (flagged in `docs/environment/current.md` lines 26 and 34). Six frontend modules read `VITE_API_URL`, four read `VITE_API_BASE_URL`. Both default to `http://localhost:8000`. Consolidate to a single canonical name `VITE_API_BASE_URL` (the more semantically correct of the two) and remove `VITE_API_URL` from the codebase. Files to update:

- `src/components/workspace/AdminConfigPanel.tsx::API_BASE_URL` (line 61)
- `src/components/workspace/AIChatInterface.tsx`
- `src/components/workspace/CaseContextDialog.tsx`
- `src/components/workspace/CaseTreeExplorer.tsx`
- `src/components/workspace/ContextHierarchyDialog.tsx::API_BASE_URL` (line 63)
- `src/lib/fileApi.ts`

The replaced lines all use the pattern `import.meta.env.VITE_API_URL || "http://localhost:8000"`; the new pattern reads `VITE_API_BASE_URL` from a single source `src/lib/apiConfig.ts` (new) so the fallback is defined once.

**2. Duplicate `ErrorResponse` schema** (flagged in `docs/api/current.md` line 1097: "Defined in: `backend/api/admin.py` and `backend/api/files.py` (two distinct definitions; identical except files.py adds `file_name`)"). Move the canonical `ErrorResponse` to a new `backend/api/_shared.py` module. The `admin.py` variant uses the base shape; the `files.py` variant becomes `FileErrorResponse(ErrorResponse)` extending it with `file_name`. Update the route declarations and `responses=` keyword arguments in both routers to point at the new types.

**3. `gemini_initialized` health-check field name** is consumed by `GET /api/chat/health`, `GET /api/search/health`, and `GET /api/validation/health`. The closed-environment deployment may run on LiteLLM (per S001-F-001), so the field name becomes misleading. Add a sibling field `llm_backend` populated from the new `LLM_BACKEND` env var. **Do not remove `gemini_initialized` in this sprint** — frontend consumers still read it; a follow-up sprint can rename. This requirement only adds the new field with the value of `get_provider().__class__.__name__`.

This requirement is intentionally scoped narrowly. Other anti-patterns the user might flag (free-form `Dict[str, Any]` responses, `caseId: "UnknownCase"` defaults, `mandatory: false` defaults that don't match the actual default folder set) are out of scope for sprint 1.

## Acceptance criteria

- A new file `src/lib/apiConfig.ts` exports `export const API_BASE_URL = import.meta.env.VITE_API_BASE_URL ?? "http://localhost:8000"` and is imported by every frontend module that previously read `VITE_API_URL` or had its own copy of the constant.
- After the refactor, `git grep "VITE_API_URL"` returns zero matches in `src/`.
- `git grep "import.meta.env.VITE_API_BASE_URL"` returns exactly one match (in `src/lib/apiConfig.ts`) and the API_BASE_URL constant is imported elsewhere from there.
- A new file `backend/api/_shared.py` exists and exports `ErrorResponse` (with `error: str` and `detail: Optional[str]`) and `FileErrorResponse(ErrorResponse)` (adds `file_name: Optional[str]`).
- `backend/api/admin.py` and `backend/api/files.py` import their `ErrorResponse` / `FileErrorResponse` from `_shared.py` and the local class definitions are removed.
- All `responses=` declarations in the affected routers reference the new types; the OpenAPI schema produced by `GET /openapi.json` shows a single `ErrorResponse` schema component (not two duplicate ones).
- `GET /api/chat/health`, `GET /api/search/health`, `GET /api/validation/health` response bodies include a new `llm_backend` field whose value is one of `"LiteLLMProvider"`, `"GeminiProvider"`, or `"unconfigured"`.
- `gemini_initialized` field still exists in the same response bodies for backward compatibility.
- `docs/environment/current.md` and `docs/api/current.md`, on next regeneration by the documenter agents, no longer flag the duplicates (this is automatic; no manual edit required to the current.md files).
- The frontend continues to function without changes — health badges still parse `gemini_initialized` and the new field is silently ignored by anything that does not consume it.

## Test cases

- **TC-S001-F-009-01** (single env var): grep for `VITE_API_URL` across `src/` returns zero matches; grep for `VITE_API_BASE_URL` returns exactly one source declaration.
- **TC-S001-F-009-02** (frontend functional): with `VITE_API_BASE_URL=http://localhost:8000`, every frontend network call observed via the dev-server proxy hits the same target as before.
- **TC-S001-F-009-03** (single ErrorResponse in OpenAPI): `GET /openapi.json` and parse `components.schemas`; there is exactly one `ErrorResponse` entry, and `FileErrorResponse` references it via `allOf`.
- **TC-S001-F-009-04** (health response shape): `GET /api/chat/health` returns a body containing both `gemini_initialized` (boolean) and `llm_backend` (one of the documented strings).
- **TC-S001-F-009-05** (no breakage of existing tests): the existing `backend/tests/` suite passes after the refactor with no test-code changes other than imports for `ErrorResponse` if any test referenced the local definition.
- **TC-S001-F-009-06** (frontend health badge): the existing health badge that reads `gemini_initialized` continues to render the green/red state based on its boolean value.
