# Centralized .env schema for closed-environment deployment

## Description

Introduce a single `.env.example` at the repository root that documents every environment variable required for the closed-environment deployment described in the sprint input. The current state has only an unmanaged `.env` file; `docs/environment/current.md` notes that no `.env.example` is present, so frontend modules and `backend/config.py` rely on hard-coded fallbacks that drift between modules (e.g., the duplicate `VITE_API_URL` / `VITE_API_BASE_URL` pair).

The new `.env.example` must group variables into the same five sections the input enumerates so the operator can fill in once and have backend, frontend and `docker-compose.yml` all consume the same values:

1. **Registry & package sources** (consumed by Docker build / Compose): `DOCKER_REGISTRY`, `NPM_REGISTRY`, `PIP_INDEX_URL`. Defaults point to public sources for local dev; production overrides at the BDOP layer point at Artifactory.
2. **LLM backend** (consumed by `backend/config.py`, the new LLM provider abstraction in F-001, and by F-002): `LLM_BACKEND` (`internal` | `external`), `LITELLM_PROXY_URL`, `LITELLM_TOKEN`, `LITELLM_MODEL`, `ENABLE_OLLAMA_CONTAINER`. The existing `GEMINI_API_KEY` remains as the `external` path's credential.
3. **Feature flags for stubbed-out subsystems** (consumed by F-004, F-005, F-006): `ENABLE_ANONYMIZATION`, `ENABLE_DOCUMENT_SEARCH`, `ENABLE_UPLOAD`. All default to `false` for sprint 1.
4. **Session lifecycle** (consumed by F-007): `SESSION_IDLE_TIMEOUT_MINUTES` (default `10`).
5. **Ports & paths** (consumed by Compose and the existing `DOCUMENTS_PATH`): `BACKEND_PORT` (default `8000`), `FRONTEND_PORT` (default `3000`), keep documenting existing `DOCUMENTS_PATH`, `LOG_LEVEL`, `IDIRS_BASE_URL`, `INIT_TEST_DOCS`, `SKIP_INTEGRATION_TESTS`.

The file lists every variable with: required/optional, default, type, one-line purpose, and which sprint requirement introduces it. Sensitive variables (`GEMINI_API_KEY`, `LITELLM_TOKEN`) are flagged with a leading `# 🔒` comment. The `.env.example` itself contains no real secrets — only placeholder strings.

This requirement does not modify `backend/config.py`; the per-feature requirements that introduce each variable are responsible for adding the matching `os.getenv` / `get_bool_env` call sites. This requirement only owns the documentation surface.

## Acceptance criteria

- A file named `.env.example` exists at the repository root and is committed to git.
- The file contains entries for every variable listed in the Description, in the order shown, grouped by the five labelled sections.
- For every entry the file states: variable name, default (or `(required)`), type, one-sentence purpose, and the introducing requirement ID (e.g., `# introduced by S001-F-001`).
- Sensitive variables are prefixed with the `# 🔒` marker.
- No real credential, token, or proxy URL appears in the file — only placeholder strings such as `replace-me`, `http://localhost:4000`, etc.
- The README pointer ("copy `.env.example` to `.env` …") is **not** owned by this requirement; S001-NFR-003's Quickstart section absorbs it.
- `docs/environment/current.md` is updated by the env-documenter on next run to drop the "no .env.example present" warning (this happens automatically once the file exists; no manual edit required).
