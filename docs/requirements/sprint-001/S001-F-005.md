# Disable IDIRS document-search and RAG via config flag with user-facing notice

## Description

The IDIRS service (external OpenSearch-backed semantic search and RAG) is reached via two FastAPI endpoints: `POST /api/idirs/search` (`backend/api/idirs.py::idirs_search`, line 50) and `POST /api/idirs/rag` (`idirs_rag`, line 84). Both proxy to `${IDIRS_BASE_URL}` (default `http://localhost:8010`, per `backend/config.py::IDIRS_BASE_URL`). The frontend invokes them from `src/components/workspace/AIChatInterface.tsx` (the slash-command `/search` and `/rag` flows).

For sprint 1 there is no IDIRS instance reachable in the closed environment. Add a feature flag `ENABLE_DOCUMENT_SEARCH` (default `false`) read by `backend/config.py`. When the flag is `false`:

- `POST /api/idirs/search` returns `503` with `ErrorResponse{error: "feature_disabled", detail: "Die Dokumentsuche ist in dieser Demo-Umgebung noch nicht verfﾃｼgbar."}`. The status code stays `503` because that is what the existing handler already emits when the service is unreachable, keeping the frontend's error-handling path identical.
- `POST /api/idirs/rag` returns `503` with the same shape.
- The frontend `AIChatInterface` slash-command handlers already catch `503` from these endpoints; they must be updated to display the `error.detail` text via the toast/chat-message path instead of a generic "service unavailable" string.
- New i18n keys `documentSearch.notImplemented` are added to `de.json` and `en.json` and used by the frontend when the response carries `error: "feature_disabled"`.

The existing handler bodies (the HTTP proxy calls to IDIRS) stay in place and are wrapped in an `if not ENABLE_DOCUMENT_SEARCH: return JSONResponse(503, ...)` early return. This matches the input directive to preserve mechanisms behind a flag.

## Acceptance criteria

- `backend/config.py` exposes `ENABLE_DOCUMENT_SEARCH = get_bool_env("ENABLE_DOCUMENT_SEARCH", default=False)`.
- `backend/api/idirs.py::idirs_search` and `idirs_rag` both check the flag at the top of the function and return `503` with an `ErrorResponse` populated as described when the flag is off; the existing proxy code path is untouched downstream.
- `GET /api/search/health` continues to report the flag state (e.g., adds `"document_search": "disabled"` to the response payload).
- `src/components/workspace/AIChatInterface.tsx` recognises `error: "feature_disabled"` in the 503 response body and renders the localized message via `addChatMessage` rather than as a generic error; the message uses the `documentSearch.notImplemented` i18n key.
- New i18n keys `documentSearch.notImplemented` exist in both locale files.
- With `ENABLE_DOCUMENT_SEARCH=false` (default), no outbound TCP connection to `IDIRS_BASE_URL` is opened during any chat-driven search.
- With `ENABLE_DOCUMENT_SEARCH=true`, the existing proxy flow against a mock IDIRS continues to work and returns `200`.
- `.env.example` (S001-D-001) documents `ENABLE_DOCUMENT_SEARCH` with default `false`.

## Test cases

- **TC-S001-F-005-01** (happy path, disabled): `POST /api/idirs/search` with `{"query": "passport"}` and `ENABLE_DOCUMENT_SEARCH=false` returns `503` with `{"error": "feature_disabled", "detail": "Die Dokumentsuche 窶ｦ"}`.
- **TC-S001-F-005-02** (RAG disabled): same as above for `POST /api/idirs/rag` with `{"doc_ids": ["x"], "question": "y"}`.
- **TC-S001-F-005-03** (no outbound traffic): under TC-01, `tcpdump` shows zero outbound connections to `IDIRS_BASE_URL`.
- **TC-S001-F-005-04** (frontend message): in `AIChatInterface`, invoking the slash-command for search while disabled inserts a chat message containing the German translation of `documentSearch.notImplemented`.
- **TC-S001-F-005-05** (enabled passthrough): with `ENABLE_DOCUMENT_SEARCH=true` and a mocked IDIRS responding `200`, `POST /api/idirs/search` returns the upstream payload verbatim, the same as the current behaviour.
- **TC-S001-F-005-06** (health endpoint): `GET /api/search/health` body includes `"document_search": "disabled"` when the flag is off.
