# German-first localization with graceful missing-translation fallback

## Description

Today both `src/i18n/locales/de.json` and `src/i18n/locales/en.json` are maintained side-by-side; the project memory in `~/.claude/projects/.../MEMORY.md` already notes "must keep both in sync". The sprint 1 deliverable is a German-first demo: every UI string the user can encounter must have a German translation, and the English translations are nice-to-have but must not break the UI when missing.

Configure `i18next` in `src/i18n/config.ts` so that:

- The default language (`fallbackLng`) is `de`.
- Missing keys do not throw or render `undefined` / the bare key in production. Instead, when the requested language is `en` and the key is absent in `en.json`, the i18next instance falls back to `de.json`. When the key is also absent in `de.json`, the fallback returns the key string itself (e.g., `"app.title"`) so a developer can spot it but the UI does not crash.
- The language picker in `src/components/workspace/WorkspaceHeader.tsx` continues to offer DE / EN; switching to a language that is missing keys triggers the fallback transparently.
- A lint script `scripts/i18n-coverage.js` (Node, no extra dependency — uses Node 18+ built-ins) parses both locale files and emits a console warning listing keys present in `de.json` but missing in `en.json`. The script exits `0` on missing English keys (warning only) and exits `1` on missing German keys (German is required).
- The script is wired into `npm run i18n:check` in `package.json`. It is **not** added to the build pipeline in this sprint to avoid breaking CI on missing English; that gating can be added later.

This requirement does not translate any new strings — it changes the framework configuration so that subsequent requirements (F-003, F-004, F-005, F-006, F-007) can introduce new German keys without simultaneously adding English ones, and the UI gracefully handles the gap.

## Acceptance criteria

- `src/i18n/config.ts` initializes i18next with `fallbackLng: "de"`, `returnNull: false`, and `parseMissingKeyHandler: (key) => key`.
- The i18next config has `debug: false` in production builds and `debug: true` only when `import.meta.env.DEV` is true.
- Switching the UI language to `en` while a key is missing in `en.json` renders the German translation.
- Switching the UI language to `en` while a key is missing in **both** locale files renders the bare key (e.g., `app.title`) in the DOM rather than `undefined`, with no console error.
- `scripts/i18n-coverage.js` exists at the project root and is executable via `node scripts/i18n-coverage.js`.
- The script reads both locale files, walks them recursively, and prints a warning per key present in `de.json` but missing in `en.json`.
- The script exits `1` if any key is missing in `de.json` that is not in `en.json` either (impossible by definition, but covers the inverse — a key only in `en.json`).
- `package.json` contains a new `"scripts": { "i18n:check": "node scripts/i18n-coverage.js" }` entry.
- Running the existing app smoke flow (`Login` → `WorkspaceHeader` → `CaseContextDialog` → `AdminConfigPanel`) in the `en` language with no missing keys renders identically to today's behaviour.
