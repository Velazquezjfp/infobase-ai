# Disable file upload via config flag with user-facing notice

## Description

The closed-environment demo of sprint 1 must not accept new document uploads. Today the upload pipeline is exercised by `POST /api/files/upload` (`backend/api/files.py::upload_file`, line 97) and triggered from the frontend in `src/components/workspace/CaseTreeExplorer.tsx::handleUploadClick` (line 206), the drag-and-drop wrapper `src/components/workspace/FileDropZone.tsx::uploadFiles` (line 189), and the helpers in `src/lib/fileApi.ts` (`uploadFile`, `uploadFileSimple`, `uploadFileWithProgress`).

Add a feature flag `ENABLE_UPLOAD` (default `false`) read by `backend/config.py`. When the flag is `false`:

- `POST /api/files/upload` returns `403` with `ErrorResponse{error: "feature_disabled", detail: "Das Hochladen ist in dieser Demo-Umgebung deaktiviert.", file_name: <provided>}`. The `403` is reused (the schema for this endpoint already lists `403` for path-traversal denial) so no new status code is added to the API surface.
- The `GET /api/files/health` response toggles its `features.upload` field from `true` (the current hard-coded value, per the example response in `docs/api/current.md`) to the actual `ENABLE_UPLOAD` boolean.
- `CaseTreeExplorer::handleUploadClick` checks the health flag (or the response of an upload attempt) and shows a toast with `upload.notImplemented` i18n string instead of opening the file picker, when the flag is off.
- `FileDropZone` ignores drop events when the flag is off and shows the same toast.
- The "Upload" button in the UI is rendered in a disabled visual state (greyed-out, `aria-disabled="true"`) when `ENABLE_UPLOAD=false`, with a tooltip pointing to the disabled-message string.

The backend handler body remains intact behind the early-return guard so re-enabling later is a single env-var flip.

## Acceptance criteria

- `backend/config.py` exposes `ENABLE_UPLOAD = get_bool_env("ENABLE_UPLOAD", default=False)`.
- `backend/api/files.py::upload_file` checks the flag at the top of the handler; returns `403` with the documented `ErrorResponse` when off.
- `backend/api/files.py::files_health_check` reflects the flag in `features.upload` (currently hard-coded to `true`).
- `src/lib/fileApi.ts::uploadFile`, `uploadFileSimple`, `uploadFileWithProgress` translate a `403` with `error: "feature_disabled"` into a typed disabled-state result rather than throwing a generic error.
- `src/components/workspace/CaseTreeExplorer.tsx::handleUploadClick` short-circuits before opening the file picker when the upload feature is disabled (determined by the cached health-check or by attempting once and caching the result).
- `src/components/workspace/FileDropZone.tsx::uploadFiles` calls a no-op disabled handler when the flag is off and surfaces the toast.
- The Upload button in the UI is rendered with `aria-disabled="true"` and `disabled` attribute when the feature is off.
- New i18n keys `upload.notImplemented` exist in both `de.json` and `en.json`.
- `.env.example` (S001-D-001) documents `ENABLE_UPLOAD` with default `false`.

## Test cases

- **TC-S001-F-006-01** (happy path, disabled): `POST /api/files/upload` with a valid multipart body and `ENABLE_UPLOAD=false` returns `403` with `error: "feature_disabled"`.
- **TC-S001-F-006-02** (no file written): under TC-01, no file appears under `${DOCUMENTS_PATH}/${case_id}/` after the request — verified by snapshotting the directory before and after.
- **TC-S001-F-006-03** (UI button disabled): with `ENABLE_UPLOAD=false`, render `CaseTreeExplorer`; the Upload button has `aria-disabled="true"` and clicking it does **not** open a file picker.
- **TC-S001-F-006-04** (drop ignored): drag-and-drop a PDF onto `FileDropZone`; the `uploadFiles` mock is not called and a toast with `upload.notImplemented` text is shown.
- **TC-S001-F-006-05** (health flag): `GET /api/files/health` returns `features.upload === false` when the flag is off, `true` when on.
- **TC-S001-F-006-06** (enabled passthrough): with `ENABLE_UPLOAD=true`, an upload of a 1KB text file under the 15MB limit returns `200` and the file persists in the documents volume — same as today's behaviour.
