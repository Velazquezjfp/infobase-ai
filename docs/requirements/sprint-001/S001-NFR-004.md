# Containerize the LiteLLM proxy as a self-contained litellm/ subproject (gitignored)

## Description

Sprint 1 makes LiteLLM the primary front door for all LLM calls (see S001-F-001). The proxy itself must run as a container on the operator's machine so it can be started/stopped independently of the application stack and its files can be regenerated locally without ever being committed to the repository.

Create a new top-level subdirectory `litellm/` at the repository root that behaves as an independent **subproject** — its contents are intentionally gitignored. The application's `docker-compose.yml` (created by S001-NFR-003) does not include the LiteLLM service; instead the operator brings the proxy up via `docker compose -f litellm/docker-compose.yml up -d` (or via the umbrella scripts described in `README.md`). The backend reaches the proxy through its host-published port.

The subproject contents:

- `litellm/Dockerfile` — thin wrapper over the upstream image `ghcr.io/berriai/litellm:main-stable` (Context7-verified stable tag — pin to a concrete release such as `v1.81.9-stable` at implementation time). The Dockerfile copies `config.yaml` to `/app/config.yaml` and sets the entrypoint command to `litellm --config /app/config.yaml --port 4000 --num_workers 1`.
- `litellm/config.yaml` — LiteLLM proxy configuration. `model_list` declares one entry whose `model_name` matches the `LITELLM_MODEL` value the backend sends (default `gemma3:12b`) and whose `litellm_params.model` is `ollama_chat/${LITELLM_OLLAMA_MODEL}` so the proxy routes to the host's Ollama. `litellm_params.api_base` is `${LITELLM_OLLAMA_HOST}` (default `http://host.docker.internal:11434`). `general_settings.master_key` is `os.environ/LITELLM_MASTER_KEY`. The model_info section sets `supports_function_calling: true` per the LiteLLM Ollama integration docs.
- `litellm/docker-compose.yml` — single service named `litellm`, image built from the local `Dockerfile`, exposes `4000:4000`, includes `extra_hosts: ["host.docker.internal:host-gateway"]` so the container can reach the host's Ollama on Linux, reads its env from `litellm/.env`, and declares a healthcheck against `GET /health/liveliness`.
- `litellm/.env.example` — template documenting `LITELLM_MASTER_KEY` (the master token the proxy enforces; default for local dev: `sk-bamf-local-dev-key`), `LITELLM_OLLAMA_HOST` (default `http://host.docker.internal:11434`), and `LITELLM_OLLAMA_MODEL` (default `gemma3:12b`). The user creates `litellm/.env` from this template; both files live inside the gitignored subproject.
- `litellm/README.md` — local-only README with the three commands the operator needs: `docker compose -f litellm/docker-compose.yml build`, `... up -d`, `... logs -f`. Documents the relationship between `LITELLM_MASTER_KEY` (this subproject) and `LITELLM_TOKEN` (read by the backend per S001-F-001) — they must be set to the same string.

`.gitignore` at the repo root gains a new entry `litellm/` so the entire subproject and any local secrets it contains stay out of git. The umbrella `README.md` gains a "Local LiteLLM proxy" section pointing at `litellm/README.md` for the regeneration steps; the section explains that the directory is intentionally not in version control and gives operators the exact file contents they need to recreate it.

The container exposes the OpenAI-compatible REST surface on `http://localhost:4000` from the host's perspective. The backend (S001-F-001) reaches the proxy via `LITELLM_PROXY_URL=http://localhost:4000` for host-run backend, or `http://host.docker.internal:4000` for the containerized backend (S001-NFR-001) running under the umbrella compose. Both paths are documented in `litellm/README.md`.

The model name is intentionally configurable via `LITELLM_OLLAMA_MODEL` so the operator can swap `gemma3:12b` for `gemma4` (or any other locally-pulled Ollama model) by editing `litellm/.env` and restarting the container. No code change is required to switch models.

## Acceptance criteria

- A new directory `litellm/` exists locally at the repository root and is referenced by a new line `litellm/` in the project's `.gitignore`.
- `git check-ignore litellm/Dockerfile` exits `0` (the file is correctly ignored).
- `litellm/Dockerfile` exists locally, is based on `ghcr.io/berriai/litellm:main-stable` (or a pinned stable tag verified via Context7), and produces a container that runs `litellm --config /app/config.yaml --port 4000`.
- `litellm/config.yaml` exists locally with a `model_list` entry whose `model_name` is `gemma3:12b` and whose `litellm_params.model` is `ollama_chat/gemma3:12b` and `litellm_params.api_base` resolves to the value of `LITELLM_OLLAMA_HOST`.
- `litellm/config.yaml` declares `general_settings.master_key: os.environ/LITELLM_MASTER_KEY` so the master token is injected from the env file at container start.
- `litellm/docker-compose.yml` exists locally and brings the `litellm` service up on port `4000` with `extra_hosts: ["host.docker.internal:host-gateway"]`.
- `litellm/.env.example` exists in the local subproject documenting `LITELLM_MASTER_KEY`, `LITELLM_OLLAMA_HOST`, and `LITELLM_OLLAMA_MODEL` with sane defaults.
- `litellm/README.md` exists in the local subproject and documents the three lifecycle commands plus the `LITELLM_MASTER_KEY ↔ LITELLM_TOKEN` consistency requirement.
- The umbrella `README.md` gains a "Local LiteLLM proxy" section that explains the gitignored subproject pattern and links to `litellm/README.md`.
- After `docker compose -f litellm/docker-compose.yml up -d`, `curl -s http://localhost:4000/health/liveliness` returns `{"status":"healthy"}` (or LiteLLM's documented healthcheck response shape) within 30 seconds of startup.
- After startup, `curl -s -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" http://localhost:4000/v1/models` returns a JSON object whose `data[].id` includes `"gemma3:12b"`.
- An end-to-end completion call against the proxy succeeds: `curl -s -H "Authorization: Bearer ${LITELLM_MASTER_KEY}" -H "Content-Type: application/json" -d '{"model":"gemma3:12b","messages":[{"role":"user","content":"say hello"}]}' http://localhost:4000/v1/chat/completions` returns a `200` with a non-empty `choices[0].message.content`.
- Editing `LITELLM_OLLAMA_MODEL` in `litellm/.env` from `gemma3:12b` to a different locally-installed Ollama model and restarting the container changes which Ollama model the proxy serves, with no edit to `config.yaml` (the env var is read via the `os.environ/` syntax) — or, if `os.environ/` substitution is not supported for `litellm_params.model`, document the requirement to also update the model name and reload, so the env-driven swap path is explicit.
- The repository, after a clean checkout of the branch on a fresh machine, does **not** include any of the files inside `litellm/` (verified by `git status` showing only the user's own creation as untracked-and-ignored).

## Test cases

- **TC-S001-NFR-004-01** (subproject ignored): after creating `litellm/Dockerfile` locally, `git status` does not list it; `git check-ignore -v litellm/Dockerfile` reports the rule from `.gitignore`.
- **TC-S001-NFR-004-02** (image builds): `docker compose -f litellm/docker-compose.yml build` exits `0` and produces an image tagged for the `litellm` service.
- **TC-S001-NFR-004-03** (proxy live): `docker compose -f litellm/docker-compose.yml up -d` followed by a health-check loop sees `GET http://localhost:4000/health/liveliness` return `200` within 30s.
- **TC-S001-NFR-004-04** (model listed): `GET http://localhost:4000/v1/models` with the correct master key returns a body where `data[].id` contains `gemma3:12b`.
- **TC-S001-NFR-004-05** (chat completion via Ollama): a `POST /v1/chat/completions` call with `model: "gemma3:12b"` returns a `200` and the upstream Ollama logs (`journalctl -u ollama` or `ollama ps`) show the corresponding inbound request.
- **TC-S001-NFR-004-06** (auth enforced): the same `/v1/chat/completions` call without an `Authorization` header returns `401`.
- **TC-S001-NFR-004-07** (model swap via env): change `LITELLM_OLLAMA_MODEL` in `litellm/.env` to a different installed Ollama model (e.g., `nomic-embed-text:latest` for an embedding sanity check, or any chat-capable installed model), restart the container, and verify the new model is the one routed by the proxy.
- **TC-S001-NFR-004-08** (backend integration): with `LLM_BACKEND=internal`, `LITELLM_PROXY_URL=http://localhost:4000`, `LITELLM_TOKEN=${LITELLM_MASTER_KEY}`, `LITELLM_MODEL=gemma3:12b`, the backend's `POST /api/admin/generate-field` returns `200` (cross-checks with TC-S001-F-001-01).
