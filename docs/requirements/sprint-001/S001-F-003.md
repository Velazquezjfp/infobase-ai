# Render dynamic-context URLs as plain text in the closed environment

## Description

The `ContextHierarchyDialog` (`src/components/workspace/ContextHierarchyDialog.tsx`) renders the active case's context tree, including any URL-typed entries that come from the case JSON files (`backend/data/contexts/cases/{caseId}/case.json` `regulations` array, the per-folder context files under `backend/data/contexts/cases/{caseId}/folders/*.json`, and any custom rules). In the closed deployment there is no internet access, so clicking those links would fail; the user wants them displayed as inert text plus an inline note that the content is offline-simulated.

Two behaviours are needed:

1. **Frontend rendering** — `ContextHierarchyDialog::renderNode` (line 325) and any other context-tree renderer that currently produces an `<a href>` for URL-shaped values must instead render the URL as a `<span>` (or markdown-rendered text) and append a small disclaimer: in German `"Offline-Modus: Inhalt wird simuliert"` / English `"Offline mode: content is simulated"`. The `i18n` keys are introduced under `context.offlineNotice` in both `de.json` and `en.json` (S001-F-008 governs the missing-translation fallback).
2. **Backend pre-extraction (optional, fallback)** — Where a case JSON or custom rule references an external URL whose content is meant to ground LLM answers (e.g., a regulation page), there must be a way to ship the extracted text alongside the URL so the LLM still receives the substance. For sprint 1 the in-scope subset is: any URL field in `case.json::regulations[*]` may carry a sibling `extractedText` field; if present, the LLM prompt builder in `backend/services/gemini_service.py::_build_prompt` (line 450) and `_load_context` (line 376) injects the `extractedText` and ignores the `url`. This is purely opt-in — sprint 1 does not crawl or extract anything; it only consumes pre-extracted text the operator hand-prepares.

The delete-and-recreate of the URL-fetching crawler is explicitly **out of scope**. The input says "You may have to create another task for a sub agent to collect the url, extract the html and only use text or markdown instead of the navigation" — that pre-extraction is a one-off operator task documented in `README.md` and will not be a runtime feature.

The fix is therefore: (a) UI no longer turns URLs into hyperlinks; (b) prompt builder consumes pre-extracted text when present.

## Acceptance criteria

- `src/components/workspace/ContextHierarchyDialog.tsx::renderNode` no longer emits `<a target="_blank">` for URL-shaped values; the URL is shown as plain text inside a `<span class="font-mono text-xs">…</span>`.
- A muted disclaimer line appears under any rendered URL: `t("context.offlineNotice")` falls back to the German string when translation is missing (per S001-F-008).
- New i18n keys `context.offlineNotice` exist in both `src/i18n/locales/de.json` and `src/i18n/locales/en.json`.
- `backend/services/gemini_service.py::_load_context` reads `regulations[*].extractedText` when present and includes the text in the prompt produced by `_build_prompt`; the URL is included in the prompt only as a citation marker (e.g., `(source: <url>)`), not as a clickable resource.
- `GET /api/context/case/{case_id}` continues to return the raw `case.json` payload unchanged — extraction logic stays server-side in the prompt builder.
- Manually opening `ContextHierarchyDialog` for `ACTE-2024-001` shows every URL as inert text; right-click → "Open link" is unavailable (no `<a>` element).
- The chat flow still uses the URL's content when querying about regulations, provided `extractedText` was supplied by the operator.

## Test cases

- **TC-S001-F-003-01** (UI): in `ContextHierarchyDialog`, given a case context whose `regulations[0].url = "https://www.gesetze-im-internet.de/aufenthg/"`, the rendered DOM contains the URL as text inside a `<span>` and **no** `<a>` element with that `href`.
- **TC-S001-F-003-02** (UI disclaimer): the disclaimer text matches the German `t("context.offlineNotice")` string when language is `de`, and the English version when language is `en`.
- **TC-S001-F-003-03** (LLM grounding): with a fixture `regulations[0] = {url: "https://example.de/x", extractedText: "Paragraph 23 says ..."}`, send a chat question about that regulation; the LLM prompt sent by `_build_prompt` contains the substring `Paragraph 23 says` (verified by injecting a fake `LLMProvider` that records its input).
- **TC-S001-F-003-04** (no extraction, graceful): with `regulations[0] = {url: "https://example.de/x"}` and no `extractedText`, the prompt contains the URL as a citation but no fetched content; no HTTP call leaves the backend.
- **TC-S001-F-003-05** (no missing-key error): when `de.json` lacks the `context.offlineNotice` key (regression check for S001-F-008), the UI renders the key name itself rather than throwing — verified by temporarily removing the key in a fixture build.
