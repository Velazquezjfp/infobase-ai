# Optional local Ollama dev container behind a Compose flag

## Description

For local development the operator typically runs LiteLLM (containerized in S001-NFR-004) against an Ollama instance that already exists on the host (the Linux dev box, where `gemma3:12b` is pre-installed). For other developers or CI runs there should be a one-shot way to spin up an in-Compose Ollama container that pulls `gemma3:12b` on first start, so the LiteLLM container can route to either the host's Ollama or the Compose-managed one without any code change.

Add an `ollama` service to the `docker-compose.yml` produced by S001-NFR-003, gated behind the `ENABLE_OLLAMA_CONTAINER` env var via Compose profiles (`profiles: ["ollama"]`). When the operator runs `docker compose --profile ollama up`, the container is built and started; otherwise it is silently skipped. The service uses the upstream `ollama/ollama` image (pulled via `${DOCKER_REGISTRY}` so the same proxy/Artifactory toggle applies), exposes port `11434` on the Compose network only (no host port binding by default to avoid collision with a host-installed Ollama), mounts a named volume `ollama-data` for model weights, and runs an entrypoint that pulls `${LITELLM_OLLAMA_MODEL:-gemma3:12b}` on startup.

When the profile is active, the LiteLLM container from S001-NFR-004 points at the Compose-internal hostname (`http://ollama:11434`) instead of the host's Ollama. The `ENABLE_OLLAMA_CONTAINER` flag is the only switch the operator touches; the LiteLLM container's `ollama_api_base` config value is read from the env var the operator sets accordingly.

The model tag default is `gemma3:12b` (already installed locally — verified via `ollama list`). The user noted that they may switch to `gemma4` (or any other Ollama-supported tag) later; this is a pure env-var change with no code impact.

## Acceptance criteria

- `docker-compose.yml` (created by S001-NFR-003) gains an `ollama` service block with `profiles: ["ollama"]` so it is inert by default.
- The service references `${DOCKER_REGISTRY}/ollama/ollama:latest` (or a pinned tag — pinned preferred; agree with operator on the tag at implementation time).
- A named volume `ollama-data` is declared at the top level and mounted at `/root/.ollama` inside the container.
- The service has an entrypoint or post-start command that runs `ollama pull ${LITELLM_OLLAMA_MODEL:-gemma3:12b}` so the first run populates the volume; subsequent runs are no-ops.
- Running `docker compose up` (no profile) does **not** start the `ollama` service; running `docker compose --profile ollama up` does.
- `README.md` gains a "Local LLM development" section that explains the two paths: (a) host-installed Ollama + host-installed LiteLLM (default), and (b) Compose-managed Ollama via `--profile ollama`.
- The `ENABLE_OLLAMA_CONTAINER` env var is documented in `.env.example` (S001-D-001) with default `false` and a reference to the profile flag.

## Test cases

- **TC-S001-F-002-01** (happy path, profile off): `docker compose up` brings up only `backend` and `frontend`; `docker compose ps` lists those two services and no `ollama` row.
- **TC-S001-F-002-02** (happy path, profile on): `docker compose --profile ollama up` brings up `backend`, `frontend`, and `ollama`; `curl http://ollama:11434/api/tags` from inside the backend container returns `200`.
- **TC-S001-F-002-03** (model pull): on the first `docker compose --profile ollama up`, the Ollama logs include `pulling manifest` for `gemma3:12b`, and on the second run the same logs show the model already present.
- **TC-S001-F-002-04** (volume persistence): stop and remove the `ollama` container, restart with `--profile ollama`; the model is still available without re-pulling (verified by `ollama list` showing `gemma3:12b`).
- **TC-S001-F-002-05** (host port not bound by default): `ss -tlnp | grep 11434` on the host returns nothing while the Compose `ollama` service is running, so a host-installed Ollama can run side by side without port conflict.
