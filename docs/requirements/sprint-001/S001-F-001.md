# Pluggable LLM backend with LiteLLM proxy and Gemini fallback

## Description

Today, every AI call flows through hard-coded calls to `google.generativeai` from three singletons: `backend/services/gemini_service.py::GeminiService`, `backend/services/validation_service.py::CaseValidationService`, and `backend/services/shacl_generator.py::SHACLGeneratorService`. All three read `GEMINI_API_KEY` directly and instantiate a Google client. For the closed-environment deployment, the only reachable LLM endpoint is a LiteLLM proxy (containerized in S001-NFR-004 and fronting a host-installed Ollama running `gemma3:12b`). The closed network has no access to Google's public Gemini endpoint.

LiteLLM is the **primary front door** for sprint 1. The Gemini direct path is preserved only as an opt-in development convenience — it is never used in parallel with LiteLLM, and switching is a single env-var toggle.

Introduce a new `backend/services/llm_provider.py` module that exposes a single async interface (`generate(prompt, **kwargs) -> str` and `generate_streaming(prompt, **kwargs) -> AsyncIterator[str]`) and dispatches to exactly one concrete implementation chosen at process start based on `LLM_BACKEND`:

- `LLM_BACKEND=internal` (default — closed-environment, the LiteLLM path): instantiate `LiteLLMProvider`, which uses the `litellm` Python SDK (newly added to `backend/requirements.txt`) configured against the OpenAI-compatible endpoint exposed by the LiteLLM proxy from S001-NFR-004. Calls use `litellm.acompletion(model=${LITELLM_MODEL}, api_base=${LITELLM_PROXY_URL}, api_key=${LITELLM_TOKEN}, messages=[...])`. Default `LITELLM_MODEL=gemma3:12b` (matches the model already installed on the host's Ollama; see `ollama list`). The Google client is **not** imported, instantiated, or contacted on this branch — verified by TC-S001-F-001-06.
- `LLM_BACKEND=external` (developer opt-in): instantiate `GeminiProvider`, which wraps the existing `google.generativeai` path. This branch reads `GEMINI_API_KEY`. The LiteLLM proxy is **not** called on this branch.

The factory `get_provider()` reads `LLM_BACKEND` once and caches the chosen provider as a module-level singleton. Unknown values raise `ValueError` at import time, not lazily on first call. There is no automatic failover or parallel call to the other backend — exclusivity is the explicit contract.

`GeminiService`, `CaseValidationService`, and `SHACLGeneratorService` are refactored to delegate every model call to `llm_provider.get_provider()` instead of touching `google.generativeai` directly. They keep their public method signatures so callers (`backend/api/chat.py`, `backend/api/validation.py`, `backend/api/admin.py`) and the WebSocket handler do not change. The `is_initialized` health-check semantics shift to "is the provider configured" — i.e., presence of `LITELLM_PROXY_URL` for `internal` or `GEMINI_API_KEY` for `external`.

Streaming responses already flow through `_generate_streaming_response` for the WS chat handler; on the LiteLLM branch the SDK's `litellm.acompletion(stream=True)` returns an async iterator of OpenAI-style chunks which is adapted to the existing chunk-yielding signature so `backend/api/chat.py::websocket_chat_endpoint` is not touched.

The class name `GeminiService` is preserved (renaming would touch dozens of import sites and is out of scope); only the internals change. A docstring flags the misnomer for a follow-up sprint.

This requirement is the prerequisite for the testing strategy described in the sprint input: "All services currently using gemini endpoint must switch and they must be tested, but only after all functionalities have been filtered down" — meaning F-004/F-005/F-006 should land first to reduce the surface that needs to be re-tested against LiteLLM, then F-001 swaps the model provider underneath what remains.

## Acceptance criteria

- A new file `backend/services/llm_provider.py` exists and exports an `LLMProvider` Protocol/ABC with `async generate(prompt, **kwargs) -> str` and `async generate_streaming(prompt, **kwargs) -> AsyncIterator[str]` methods.
- Two concrete implementations are present: `LiteLLMProvider` (uses the `litellm` Python SDK's `acompletion`) and `GeminiProvider` (wraps the existing `google.generativeai` path).
- `litellm` is added to `backend/requirements.txt` (latest stable; resolved via Context7 at implementation time).
- A factory function `get_provider() -> LLMProvider` reads `LLM_BACKEND` from env once and returns a cached singleton; the values `internal` and `external` are accepted; any other value raises `ValueError` at import time. No fallback or parallel call to the unselected backend ever occurs.
- When `LLM_BACKEND=internal`, the `google.generativeai` module is not imported (verified by TC-06).
- When `LLM_BACKEND=external`, no HTTP request is made to `LITELLM_PROXY_URL`.
- `backend/services/gemini_service.py::GeminiService.generate_response` and `_generate_streaming_response` are refactored to call `get_provider()` instead of `google.generativeai` directly.
- `backend/services/validation_service.py::CaseValidationService.validate_case` is refactored similarly.
- `backend/services/shacl_generator.py::SHACLGeneratorService.parse_nl_command` and `apply_modification` are refactored similarly.
- `backend/config.py` exposes `LLM_BACKEND`, `LITELLM_PROXY_URL`, `LITELLM_TOKEN`, `LITELLM_MODEL` constants via `os.getenv`, with the same fallback-to-default pattern used for existing variables. Default `LITELLM_MODEL=gemma3:12b`.
- The `GET /api/chat/health`, `GET /api/search/health`, and `GET /api/validation/health` endpoints continue to return `200` when configured; the new `llm_backend` field added by S001-F-009 reflects the active provider class name.
- Switching `LLM_BACKEND` from `external` to `internal` and back via `.env` produces working chat responses against each backend without code changes.
- Streaming chat over `WS /ws/chat/{case_id}` works under `LLM_BACKEND=internal` — the client receives `chat_chunk` messages with `is_complete=true` on the final chunk.

## Test cases

- **TC-S001-F-001-01** (happy path, internal/LiteLLM): with `LLM_BACKEND=internal`, `LITELLM_PROXY_URL` pointing at the local LiteLLM container from S001-NFR-004, and `gemma3:12b` available in the host's Ollama, send `POST /api/admin/generate-field` with `{"prompt": "Add a marital-status dropdown"}`; the response is `200` with a non-empty `field` object.
- **TC-S001-F-001-02** (happy path, external/Gemini): with `LLM_BACKEND=external` and a valid `GEMINI_API_KEY`, the same request returns `200` and the chat WebSocket round-trip still streams.
- **TC-S001-F-001-03** (config error): with `LLM_BACKEND=foobar`, the application fails to start with a `ValueError` at import time; uvicorn does not begin listening.
- **TC-S001-F-001-04** (network error, internal): with `LLM_BACKEND=internal` but `LITELLM_PROXY_URL` pointing at an unreachable host, `POST /api/admin/generate-field` returns `500` with a clear `ErrorResponse.detail` mentioning the LLM provider, not a stack trace, and **no fallback call to the Gemini API is attempted** (verified by network mock).
- **TC-S001-F-001-05** (streaming): with `LLM_BACKEND=internal`, open `WS /ws/chat/{case_id}` and send a `chat` message; the client receives at least 2 `chat_chunk` messages and a final one with `is_complete=true`.
- **TC-S001-F-001-06** (no Google import on internal path): `python3 -c "import os; os.environ['LLM_BACKEND']='internal'; import backend.services.llm_provider as p; p.get_provider()"` does not import `google.generativeai`; verified by inspecting `sys.modules` after the call.
- **TC-S001-F-001-07** (no LiteLLM proxy call on external path): with `LLM_BACKEND=external` and `LITELLM_PROXY_URL=http://127.0.0.1:1` (unreachable), the application starts and `POST /api/admin/generate-field` succeeds against Gemini — proving the LiteLLM URL was never contacted.
