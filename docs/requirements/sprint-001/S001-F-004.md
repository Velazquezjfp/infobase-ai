# Disable anonymization service via config flag with user-facing notice

## Description

The anonymization pipeline currently lives in `backend/services/anonymization_service.py::AnonymizationService` (singleton) and `backend/tools/anonymization_tool.py::anonymize_document`. It is invoked from the WebSocket handler `backend/api/chat.py::handle_anonymization` (line 472) when the client sends a `{"type": "anonymize", ...}` message. The service depends on an external HTTP API (its URL and secret are passed at instantiation, per the existing test `test_api_url_configuration`), which is unreachable in the closed environment of sprint 1.

Add a feature flag `ENABLE_ANONYMIZATION` (default `false`) read by `backend/config.py` via the existing `get_bool_env` helper. When the flag is `false`:

- `handle_anonymization` short-circuits and emits an `anonymization_complete` WebSocket message with `success: false`, `error: "feature_disabled"`, and a localized user-facing message (`anonymization.notImplemented` i18n key, German: `"Die Anonymisierungsfunktion ist in dieser Demo-Umgebung noch nicht verfügbar."` / English: `"The anonymization feature is not yet available in this demo environment."`).
- `AnonymizationService.check_service_health` returns immediately with `{"status": "disabled"}` and does not attempt any outbound HTTP request.
- The frontend `CaseTreeExplorer` and any other UI that triggers anonymization receives the disabled-message and displays it via the existing toast/notification path; no error toast is shown.

Critically, **the existing service code stays in place** — only the entry points are gated. The input is explicit: "do not get rid of the current mechanisms, simply use a variable in config file to use connections or not, at first I need to disable them, after at some point I will actually implement them".

The integration test `backend/tests/test_anonymization.py::TestAnonymizationIntegration` already gates itself on `SKIP_INTEGRATION_TESTS=1`. No change is required to that file other than making sure the new flag does not cause unit tests to fail (unit tests should set `ENABLE_ANONYMIZATION=true` before importing or use the singleton's existing API directly).

## Acceptance criteria

- `backend/config.py` exposes a constant `ENABLE_ANONYMIZATION = get_bool_env("ENABLE_ANONYMIZATION", default=False)`.
- `backend/api/chat.py::handle_anonymization` reads the flag at the top of the function body; if `False`, it sends an `anonymization_complete` message with `success=false`, `error="feature_disabled"`, and a `message` field populated from a backend constant that mirrors the i18n string.
- The WebSocket handler does not invoke `AnonymizationService.anonymize_image` when the flag is off.
- New i18n keys `anonymization.notImplemented` exist in both `de.json` and `en.json` and are displayed when the frontend receives an `error="feature_disabled"` anonymization response.
- With `ENABLE_ANONYMIZATION=false` (default), the backend never opens an outbound TCP connection to the configured anonymization API URL during a chat session — verified via `tcpdump`/log inspection in the test plan.
- With `ENABLE_ANONYMIZATION=true`, the existing flow is unchanged and the legacy tests under `backend/tests/test_anonymization.py` still pass.
- `.env.example` (S001-D-001) lists `ENABLE_ANONYMIZATION` with default `false` and the introducing requirement marker.

## Test cases

- **TC-S001-F-004-01** (happy path, disabled): with `ENABLE_ANONYMIZATION=false`, send `{"type": "anonymize", "filePath": "x.png", "documentId": "1", "folderId": "personal-data"}` over `WS /ws/chat/{case_id}`; the server replies with `{"type": "anonymization_complete", "success": false, "error": "feature_disabled", "message": "Die Anonymisierungsfunktion …"}` within 200ms.
- **TC-S001-F-004-02** (no outbound traffic): under TC-01 conditions, `tcpdump` on the backend container shows zero outbound connections to the configured anonymization API host.
- **TC-S001-F-004-03** (frontend toast): the React UI receives the disabled message and displays an info toast (not error); the toast text matches the language-active translation.
- **TC-S001-F-004-04** (enabled path unchanged): with `ENABLE_ANONYMIZATION=true` and a mock anonymization API, the existing `test_full_anonymization_workflow` test passes.
- **TC-S001-F-004-05** (health endpoint): `GET /api/chat/health` reflects the flag — when disabled the response includes `"anonymization": "disabled"` (or a comparable explicit field) so the frontend can render the right state without inferring from a 503.
