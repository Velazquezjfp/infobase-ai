# Docker Compose orchestration for backend + frontend with persistent volumes

## Description

Add a `docker-compose.yml` at the project root that orchestrates the two images produced by S001-NFR-001 (`bamf-backend`) and S001-NFR-002 (`bamf-frontend`). The Compose file is the single entry point an operator uses to bring up the demo on a developer laptop or on the BDOP-equivalent dev server. It is the source of truth for: which env vars get propagated to which container, which named volumes persist what, and which ports the host exposes.

Top-level structure:

- **services**: `backend`, `frontend`, and (per S001-F-002, behind `profiles: ["ollama"]`) `ollama`.
- **volumes** (named): `documents-data` (mounted into backend at `/var/app/documents`), `cases-data` (mounted into backend at `/app/backend/data/contexts/cases`), `manifest-data` (mounted into backend at `/app/backend/data` so `document_manifest.json` survives restarts), `ollama-data` (per S001-F-002).
- **networks** (default): single bridge network so backend and frontend can reach each other by service name.

Critical Compose elements:

- `services.backend.build`: `{ context: ., dockerfile: backend/Dockerfile, args: { DOCKER_REGISTRY: "${DOCKER_REGISTRY}", PIP_INDEX_URL: "${PIP_INDEX_URL}" } }`.
- `services.backend.env_file: [".env"]` so every consuming env var (LLM_*, ENABLE_*, IDIRS_*, GEMINI_API_KEY, DOCUMENTS_PATH, LOG_LEVEL, SESSION_IDLE_TIMEOUT_MINUTES) flows in from a single file.
- `services.backend.ports: ["${BACKEND_PORT:-8000}:8000"]`.
- `services.backend.volumes`: the three named volumes above mounted at the documented paths.
- `services.backend.deploy.resources.limits: { cpus: "2.0", memory: 2048M }` (informational on Compose v3.9; honoured on Swarm; documented in `README.md`).
- `services.backend.healthcheck`: same `/health` curl as the Dockerfile, with `interval: 30s`, `timeout: 5s`, `retries: 3`, `start_period: 15s`.
- `services.backend.restart: unless-stopped`.
- `services.frontend.build`: `{ context: ., dockerfile: frontend/Dockerfile, args: { DOCKER_REGISTRY: "${DOCKER_REGISTRY}", NPM_REGISTRY: "${NPM_REGISTRY}", VITE_API_BASE_URL: "${VITE_API_BASE_URL:-http://localhost:8000}" } }`.
- `services.frontend.ports: ["${FRONTEND_PORT:-3000}:3000"]`.
- `services.frontend.depends_on.backend.condition: service_healthy` so the frontend waits for the backend's health check to flip green before starting.
- `services.frontend.deploy.resources.limits: { cpus: "1.0", memory: 1024M }`.
- `services.frontend.restart: unless-stopped`.

The Compose file uses the v3.9 schema (`version: "3.9"` literal). It does not depend on Compose-spec features that require Compose v2.20+; it works with Compose v2.5 and later.

The hard rule for sprint 1: every value that differs between dev and prod **must** come from `.env` (the file the operator copies from `.env.example`, per S001-D-001). The Compose file itself contains zero hard-coded URLs, tokens, or paths — only fallbacks via the `${VAR:-default}` shell-substitution syntax for things that are safe to default in dev.

## Acceptance criteria

- A file `docker-compose.yml` exists at the project root with `version: "3.9"`.
- `docker compose config` (validation) passes with no warnings about deprecated keys.
- `docker compose up -d` brings up `backend` and `frontend`; `docker compose ps` lists both as `healthy` within 60 seconds.
- `curl http://localhost:8000/health` returns `200`.
- `curl http://localhost:3000/` returns `200`.
- The frontend, loaded in a browser, makes `/api/...` calls that land on the backend (verified by tailing `docker compose logs backend`).
- `docker volume ls` shows the four named volumes (`documents-data`, `cases-data`, `manifest-data`, plus `ollama-data` after running with the `ollama` profile).
- Stopping and recreating the backend container preserves uploaded files (with `ENABLE_UPLOAD=true`) — verified by uploading a file, `docker compose down`, `docker compose up -d`, and confirming the file is still listed in `GET /api/documents/all`.
- `docker compose --profile ollama up -d` additionally starts the `ollama` service (S001-F-002 governs that block; this requirement only ensures the Compose file is shaped to accept the profile addition).
- The Compose file contains no hard-coded secret values (grep returns empty for `AIza`, `sk-`, real proxy URLs).
- `README.md` gains a "Quickstart" section describing: copy `.env.example` to `.env`, edit `LITELLM_*` and `GEMINI_API_KEY`, `docker compose up -d`, browse to `http://localhost:3000`.

## Test cases

- **TC-S001-NFR-003-01** (validation): `docker compose config` exits `0` and prints a normalized config containing `services.backend` and `services.frontend`.
- **TC-S001-NFR-003-02** (up + health): `docker compose up -d --build && sleep 60 && docker compose ps --format json | jq '.[] | select(.Service=="backend").Health'` returns `"healthy"` and the same for `frontend`.
- **TC-S001-NFR-003-03** (volume persistence): start, `POST /api/folders/ACTE-2024-001` to create a new folder, `docker compose down`, `docker compose up -d`, `GET /api/folders/ACTE-2024-001` shows the new folder still present (because `cases-data` volume persists).
- **TC-S001-NFR-003-04** (env propagation): set `LOG_LEVEL=DEBUG` in `.env`, `docker compose up -d`, `docker compose exec backend env | grep LOG_LEVEL` shows `LOG_LEVEL=DEBUG`.
- **TC-S001-NFR-003-05** (frontend reaches backend): browse to `http://localhost:3000`, observe at least one `200` response in `docker compose logs backend` for an `/api/` path within 10 seconds.
- **TC-S001-NFR-003-06** (no secrets in repo): `git diff` of `docker-compose.yml` against an empty file contains no token-shaped substrings (grep `AIza`, `sk-`, `Bearer ` returns nothing).
- **TC-S001-NFR-003-07** (profile inert): `docker compose up -d` without `--profile ollama` does not start the `ollama` service.
- **TC-S001-NFR-003-08** (depends_on): with `services.frontend.depends_on.backend.condition: service_healthy`, killing the backend's `/health` (e.g., `docker compose pause backend`) and restarting causes the frontend to wait until backend health flips green.
