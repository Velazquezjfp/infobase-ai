## 2026-04-28

### Modified
- `LITELLM_PROXY_URL`: required-status semantics tightened. Config-layer default unchanged (`''` via `os.getenv('LITELLM_PROXY_URL', '')`), but `backend/services/llm_provider.py::_build_provider` (added by S001-NFR-004) now raises `ValueError("LITELLM_PROXY_URL is required when LLM_BACKEND=internal — start the LiteLLM container per S001-NFR-004 or set LLM_BACKEND=external")` before any provider is instantiated when `LLM_BACKEND=internal` and the URL is empty/blank. Previously the failure surfaced silently at first request time. Doc entry now also lists `backend/services/llm_provider.py::_build_provider` under "Used by".
- `GEMINI_API_KEY`: doc entry sharpened — explicit `ValueError("GEMINI_API_KEY not configured for external backend")` from `GeminiProvider.__init__` (S001-F-001 behaviour, previously documented loosely as "the Gemini provider will fail to initialise"). No code change in this wave; documentation parity with the `LITELLM_PROXY_URL` precondition.
- `LITELLM_TOKEN`: doc entry now cross-references `litellm/.env::LITELLM_MASTER_KEY` (introduced by S001-NFR-004) — the two values must match for the proxy to authenticate backend requests. No code change; documentation linkage only.
- `LLM_BACKEND`: example value in doc updated from `external` to `internal` to match the `.env.example` cutover landed by S001-NFR-004 (the safe default is now the closed-environment path because the proxy ships as a local container). Code-side default in `backend/config.py` was already `internal` since S001-F-001; no code change in this wave.
